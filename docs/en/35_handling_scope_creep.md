# Handling Scope Creep

Scope creep occurs when new requirements, features, or requests are added to a project after its scope has already been defined, without adjustments to time, budget, or resources. As an IT professional, you need to handle this by highlighting trade-offs, discussing the impact on the timeline, and guiding stakeholders to focus on the original goals.

## 1. Recognizing and Flagging Scope Creep

When a stakeholder tries to sneak in a new feature, point out politely but clearly that it is a change in the original agreement.

- "That is a great feature idea, but it falls outside the original scope of our MVP."
- "This request introduces new requirements that were not part of the initial specification."
- "Adding this payment method would be a significant change to the current sprint backlog."

## 2. Explaining the Impact on Timeline & Resources

Help stakeholders understand that adding features is not free; it costs time and resources.

- "Adding this reporting dashboard now will delay our target release date by at least a week."
- "If we implement this custom user group management, we won't have enough time to complete the core payment flow in this sprint."
- "Our team is currently at 100% capacity. To take on this new task, we would need to push back another priority item."

## 3. Proposing a Phase-Based Approach

Instead of saying a flat "no," suggest doing it in a future phase. This keeps the relationship positive.

- "I suggest we put this request in our product backlog and prioritize it for Phase 2."
- "Can we launch the core features first, and then add this enhancement in a fast-follow update?"
- "Let's capture this requirement in a separate Jira ticket so we don't lose track of it, but keep our focus on the current release."

## 4. Initiating the Change Control Process

If the new feature is absolutely necessary, explain that the project plan needs to be officially updated.

- "If this is a must-have for the upcoming launch, we will need to adjust our release date."
- "To accommodate this request, we should run a formal estimation and update the project proposal."
- "Let's discuss this in a change-control meeting this Friday to decide how to adjust the budget and schedule."

## 5. Templates for Pushing Back on Creeping Scope

- **When a PM asks for a "quick change":** "Although it looks like a small change on the frontend, it requires major changes to our database schema. I recommend we defer this until after our initial release."
- **When a client adds features mid-sprint:** "We are currently in the middle of our sprint, and our scope is locked. Let's discuss this during the next sprint planning meeting."
- **When the requirements change completely:** "Given that these requirements are substantially different from the original plan, we need to pause development and re-estimate the entire module."
