# English Speaking Grammar Reference

## Practical foundation, modals, prepositions, and softening patterns for developer communication

> **Purpose of this document**
> - A focused reference for the core foundations of speaking grammar.
> - Covers weekly meetings, daily standups, Q&A with clients, follow-ups, and logic explanations.
> - Prioritizes short, clear, natural, and easy-to-remember patterns.
> - **Note:** For deep dives and detailed rules on tenses, relative clauses, parts of speech, conjunctions, comparatives, gerunds, and conditionals, please check the **Speaking Grammar Hacks** category in the sidebar!

# 1. GRAMMAR MINDSET FOR SPEAKING

## 1.1. The Real Goal
When speaking in meetings, you don't need complex grammar. You just need:
- **Correct main idea**: Focus on clarity over complexity.
- **Correct basic tense**: Choose the right tense so the client knows when events occurred.
- **Short, easy-to-understand sentences**: Avoid long running sentences. Split ideas with periods.
- **Repeat familiar formulas**: Use templates to reduce thinking time.

## 1.2. The Safest Formulas
If stuck, always fall back to these:
- **We completed [task].**
- **We're working on [task].**
- **We'll focus on [task].**
- **We found an issue with [topic].**
- **Could you clarify [point]?**
- **Let me double-check and get back to you.**

# 2. BASIC SENTENCE STRUCTURES

- **Structure 1 (S-V):** **The system works.** / **The API failed.**
- **Structure 2 (S-Be-Adj):** **The feature is ready.** / **The server is down.**
- **Structure 3 (S-V-O):** **We finished the API integration.** / **The team reviewed the feedback.**
- **Structure 4 (S-V-That-clause):** **We believe that this approach is safer.**
- **Structure 5 (If + Condition, Result):** **If the user clicks this, the modal opens.**

# 3. MODALS & DEGREES OF CERTAINTY

In client communication, you want to avoid promising things you aren't sure about. Use modals to manage expectations.

- **will** (100%): Sure commitment.
  - **"We will fix this by tomorrow."** (We promise to do it).
- **should** (80%): Expected outcome based on normal logic.
  - **"The server should restart automatically."** (If things go as planned).
- **may / can** (50%): Possibility.
  - **"This issue may delay the release."** (It is possible).
- **might / could** (20%): Cautious possibility, low certainty.
  - **"We could run into database issues if we change this configuration."**

# 4. DIPLOMATIC PASSIVE VOICE

Use passive voice to sound professional, objective, or to **soften blame** when explaining mistakes or issues.
- **Active (Direct/Blaming)**: "Our developer deleted the database table."
- **Passive (Diplomatic/Objective)**: "The database table was accidentally deleted."

### Common Passive Patterns
- **The issue has been resolved.** (Focus is on the fix, not who fixed it).
- **The feature was deployed yesterday.**
- **The database credentials were changed.**
- **You will be notified once the build completes.**

# 5. BUSINESS PREPOSITIONS BY DOMAIN

Prepositions can be tricky in IT. Memorize these specific collocations:

### 📱 On (Interfaces, Screens, Platforms)
- **on** the screen / page / dashboard
- **on** mobile / desktop
- **on** AWS / the cloud
- **on** the homepage

### 💾 In (Code, Database, Status)
- **in** the code / script
- **in** the database / table
- **in** the database config
- **in** progress / in review

### 🌐 At (End-points, Specific spots)
- **at** the endpoint URL
- **at** the bottom of the page
- **at** the moment

### ⏳ By (Deadlines)
- **by** Friday / tomorrow / 5 PM

# 6. SOFTENING & DIPLOMACY PHRASES

How to sound polite even when saying "no" or delivering bad news.

### 6.1. Disagreeing Politely
- **"I see your point, but..."**
- **"That is a good option, however, it might take longer to build."**

### 6.2. Softening Requests
- **"Could you please confirm the priority?"**
- **"It would be helpful if you could share the design specs."**
- **"Would you mind double-checking this logic?"**

### 6.3. Saying "I don't know" Safely
- **"I don't have the exact answer right now. Let me check and get back to you."**
- **"I want to verify this with the team first to give you the most accurate response."**

# 7. COMMON MISTAKES TO AVOID

### ❌ Mistake 1: "Discuss about"
- _Wrong_: "We need to discuss about the database design."
- _Correct_: **"We need to discuss the database design."**

### ❌ Mistake 2: "Explain me"
- _Wrong_: "Can you explain me this feature?"
- _Correct_: **"Can you explain this feature to me?"** / **"Can you explain this feature?"**

### ❌ Mistake 3: Forgetting past tense for done tasks
- _Wrong_: "Yesterday we deploy the website."
- _Correct_: **"Yesterday we deployed the website."**

### ❌ Mistake 4: "I'm agree"
- _Wrong_: "I'm agree with you."
- _Correct_: **"I agree with you."**

# 8. QUICK REFERENCE CHEAT SHEET

## 8.1. Status Reporting Cheat Sheet
- **Finished Task**: **"We have completed [task]."** / **"We completed [task] yesterday."**
- **In-Progress Task**: **"We are currently working on [task]."** / **"We're working on [task]."**
- **Next Steps**: **"Next, we will focus on [task]."** / **"We plan to start [task] tomorrow."**
- **Blockers**: **"We ran into a blocker with [topic]."** / **"We have an issue with [topic]."**

## 8.2. Preposition Table

| Phrase | Correct Preposition | Example |
| :--- | :--- | :--- |
| Work | **on** a task | "We are working **on** the payment gateway." |
| Focus | **on** an issue | "Let's focus **on** the performance bug." |
| Look | **into** a bug | "We will look **into** the connection error." |
| Follow up | **on** a thread | "I'll follow up **on** that email today." |
| Agree | **with** a person / **on** a topic | "We agree **on** the proposed timeline." |
| Issue | **with** a feature | "There is an issue **with** the login form." |
